# ドラフト: 一覧画面とカテゴリ表現の設計

## 背景

例: **Mt. Fuji Trail 100** という大会名の下に、実際には **100mi / 70k / 40k / Sakuya** といった複数カテゴリがある。

- 大会単位: 日付、場所、アクセス、天気など
- カテゴリ単位: 距離、獲得標高、制限時間、スタート時間、カットオフタイム、申込費用など

詳細ページはカテゴリ単位の情報が中心。一覧画面で「70k のレースを探したい」「制限時間 24時間以上のレースを探したい」といった検索をどう実現するか、設計が未定。

---

## 1. 大会名とカテゴリの表現

### 1.1 データモデル（現状）

```
events (大会)
  └── categories (カテゴリ) 1:N
```

- `events.name`: 大会名（例: "Mt. Fuji Trail 100"）
- `categories.name`: カテゴリ名（例: "100mi", "70k", "40k", "Sakuya"）

### 1.2 表示の考え方

| 場面 | 表現例 |
|------|--------|
| 一覧 | 大会名 + カテゴリ一覧 | 「Mt. Fuji Trail 100」<br><small>100mi / 70k / 40k / Sakuya</small> |
| 詳細 | 大会名 + カテゴリ別セクション | 「Mt. Fuji Trail 100」<br>→ 各カテゴリ（100mi, 70k, 40k, Sakuya）ごとに距離・獲得標高・制限時間・スタート・カットオフなどを表示 |
| 検索結果 | 大会 + 該当カテゴリ | 「70k で検索」→ 該当する大会と、その中の 70k カテゴリを強調 |

### 1.3 カテゴリ名の正規化（任意）

- 100mi / 70k / 40k などは `distance_km` と整合する想定
- カテゴリ名は公式表記をそのまま使う（例: "Sakuya"）

## 2. 一覧画面の設計案

### 2.1 検索・絞り込みの軸

| 軸 | スコープ | 例 |
|----|----------|-----|
| 開催日 | 大会 | 2026年5月 |
| レース種別 | 大会 | トレラン / マラソン / スパルタン |
| ステイタス | 大会 | 日帰り可能 / 前泊必須 / 後泊推奨 |
| **距離** | **カテゴリ** | 40k〜70k、100k以上 |
| **獲得標高** | **カテゴリ** | 2000m以上、5000m以上 |
| **制限時間** | **カテゴリ** | 12時間以上、24時間以上 |
| **スタート時間** | **カテゴリ** | 早朝（5時台）など |

### 2.2 一覧の表示単位（2案）

#### 案A: 大会単位カード（推奨）

- **1カード = 1大会**
- カード内にカテゴリをチップまたはバッジで表示
- 絞り込み: 「距離 40k〜70k」→ 該当カテゴリを持つ大会のみ表示

```
┌─────────────────────────────────────────────────────────┐
│ Mt. Fuji Trail 100                    2026-05-10 山梨   │
│ [100mi] [70k] [40k] [Sakuya]    前泊必須  トレラン      │
│ 申込: 11〜1月  約800人  ← 大会概要                         │
└─────────────────────────────────────────────────────────┘
```

- カテゴリチップにホバーで距離・獲得標高・制限時間をツールチップ表示
- クリック → 詳細ページ（該当カテゴリへスクロールは将来対応）

#### 案B: 大会×カテゴリ単位（行を増やす）

- **1行 = 1大会×1カテゴリ**
- 「Mt. Fuji Trail 100 - 70k」「Mt. Fuji Trail 100 - 40k」のように展開

```
┌─────────────────────────────────────────────────────────┐
│ Mt. Fuji Trail 100 - 70k    70km D+3500 制限24h  山梨   │
└─────────────────────────────────────────────────────────┘
┌─────────────────────────────────────────────────────────┐
│ Mt. Fuji Trail 100 - 40k    40km D+2000 制限12h  山梨   │
└─────────────────────────────────────────────────────────┘
```

- カテゴリ条件で絞り込みやすい
- 同一大会が複数行になり、冗長になる可能性

### 2.3 推奨: 案A + カテゴリ条件フィルタ

- 一覧は **大会単位** で表示
- フィルタに **距離・獲得標高・制限時間** を追加
- 検索条件: 「その大会に、条件を満たすカテゴリが1つ以上あるか」で判定

```
┌─ フィルタ ───────────────────────────────────────────────┐
│ レース種別 [すべて▼] ステイタス [すべて▼] 開催月 [____]   │
│ 距離 [40k〜70k▼] 獲得標高 [2000m以上▼] 制限時間 [12h以上▼] │
└─────────────────────────────────────────────────────────┘
```

- カード内のカテゴリチップで、検索条件にマッチしたものを強調（オプション）

## 3. 詳細ページの構成（現状の整理）

- 大会単位: 基本情報、天気、申込み、アクセス、宿泊、その他
- **カテゴリ単位**: 各カテゴリをカードで表示
  - 距離、獲得標高、制限時間（上部に強調）
  - スタート時間、受付終了、受付場所、スタート場所
  - 完走率、カットオフタイム、必要ペース、必携品、申込費用 など

→ この構成は維持。カテゴリ単位の情報が中心であることが明確。

---

## 4. 実装ステップ（案）

1. **シード**: Mt. Fuji Trail 100 に 100mi / 70k / 40k / Sakuya を追加
2. **一覧**: カテゴリチップ表示、カテゴリ条件フィルタ（距離・獲得標高・制限時間）
3. **詳細**: URL でカテゴリ指定可能にする（例: `/events/:id#70k`）→ 該当カテゴリへスクロールは将来対応
4. **一覧**: カテゴリチップのホバーでサマリ表示

---

## 5. オープンな点

- カテゴリの並び順: 距離順？公式順？`sort_order` カラムを追加するか
- カテゴリ名の多言語: 現状は日本語・英語混在で許容
- 検索の UX: フィルタが多いと複雑になる可能性。段階的に追加するか、AI チャットで自然言語検索に寄せるか
