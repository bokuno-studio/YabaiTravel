# 開発ワークフロー定義（汎用テンプレート）

このファイルは Claude Code に読ませることで、以下の開発ワークフローを適用するためのテンプレートです。
プロジェクトの CLAUDE.md にコピーするか、このファイルを参照させてください。

---

## 基本方針

- タスクはすべて GitHub Issue で管理する（ラベル駆動）
- ユーザーの承認なしに実装は行わない
- デプロイはステージング確認 → 本番の順で必ず行う

---

## ワークフロー全体像

```
[ユーザーがやりたいことを伝える]
        ↓
[チケット作成 + pending-review]
        ↓
[ユーザーが内容確認 → 「問題ない」]
        ↓
[approved に変更]
        ↓
[ユーザーが「#XX 実装して」と指示]
        ↓
[実装 + 単体テスト → staging デプロイ → デプロイ完了監視]
        ↓
[staging-deployed に変更]
        ↓
[ユーザーがステージングで確認 → 「問題ない」]
        ↓
[staging-ok に変更]
        ↓
[ユーザーが「#XX main に出して」と指示]
        ↓
[main デプロイ → デプロイ完了監視 → エラーなし確認]
        ↓
[done を付与してクローズ]
```

---

## ラベル定義

以下のラベルを GitHub リポジトリに事前作成しておくこと。

| ラベル | 色（推奨） | 意味 | 遷移トリガー |
|--------|-----------|------|-------------|
| `pending-review` | 黄 `#FBCA04` | チケット作成済み・ユーザーレビュー待ち | チケット作成時に付与 |
| `approved` | 緑 `#0E8A16` | 内容確認OK・実装指示待ち | ユーザーが「問題ない」等と言った時 |
| `in-progress` | 青 `#1D76DB` | 実装中 | ユーザーが「#XX 実装して」と言った時 |
| `staging-deployed` | 水色 `#0075ca` | ステージングデプロイ完了・ユーザー確認待ち | staging デプロイ完了後 |
| `staging-ok` | 黄緑 `#e4e669` | ステージング確認OK・main デプロイ待ち | ユーザーが staging 確認OKを伝えた時 |
| `done` | グレー `#EDEDED` | 完了 | main デプロイ完了確認後にクローズ時付与 |

ラベル一括作成コマンド:
```bash
gh label create "pending-review" --description "レビュー待ち" --color "FBCA04"
gh label create "approved"       --description "承認済み" --color "0E8A16"
gh label create "in-progress"    --description "実装中" --color "1D76DB"
gh label create "staging-deployed" --description "ステージングデプロイ完了・ユーザー確認待ち" --color "0075ca"
gh label create "staging-ok"     --description "ステージング確認OK・mainデプロイ待ち" --color "e4e669"
gh label create "done"           --description "完了" --color "EDEDED"
```

## 各ステップの詳細ルール

### 1. チケット作成（→ pending-review）

- ユーザーがやりたいことを伝えたら、内容を整理して GitHub Issue を作成する
- まとめて伝えられた場合も、五月雨に伝えられた場合も、チケットにまとめる
- 作成時は必ず `pending-review` ラベルを付与する
- ユーザーに「以下のチケットを確認してください」と伝える

### 2. 承認（→ approved）

- ユーザーが「問題ない」「OK」「承認」等と言ったチケットを `approved` に変更する
- Claude からは `approved` に変更しない（ユーザー承認が必要）

### 3. 実装（→ in-progress → staging-deployed）

- ユーザーが「#XX 実装して」と指定したら実装を開始する
- `approved` ラベルがなくても、ユーザーからの明示的な実装指示があれば実装してよい
- 実装指示があっても該当チケットが存在しない場合は、先にチケットを作成してから実装する
- 実装開始時に `in-progress` に変更する
- 実装完了後、単体テストを実行してすべて通ることを確認する
- **フロントエンド変更の場合**: `staging` ブランチにデプロイし、デプロイ完了まで監視する
- **バックエンド変更の場合**: `--limit 3` 等で件数を絞って実際に実行し、ターミナル出力と DB 結果をチャットに貼り付けてユーザーが目視確認する
- 確認完了後 `staging-deployed` に変更し、ユーザーに報告する

### 4. ステージング確認（→ staging-ok）

- ユーザーがステージング環境を確認し、「問題ない」「OK」等と伝えたら `staging-ok` に変更する

### 5. 本番デプロイ（→ done）

- ユーザーが「#XX main に出して」「本番に出して」と指示したら main にデプロイする
- デプロイ完了・エラーなし確認後、`done` を付与してチケットをクローズする
- 「デプロイして」「プッシュして」だけでは main には出さない。必ず明示的な本番指示を待つ

---

## Claude の行動原則

- **勝手に実装しない**: ユーザーの明示的な指示なしに実装・デプロイを行わない
- **勝手にクローズしない**: フロー完了前にチケットをクローズしない
- **ラベルを正確に管理する**: 各ステップの開始・完了時に必ずラベルを更新する
- **デプロイを監視する**: デプロイ完了の確認なしに次のステップに進まない
