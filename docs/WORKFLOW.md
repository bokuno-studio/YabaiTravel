# YabaiTravel 開発ワークフロー

## 基本方針

**「意識の外で進める」** — 必要なタスクは自律的にチケット化し、レビュー完了後に実装する。

---

## チケット駆動フロー

```
[タスク特定] → [チケット作成] → [レビュー依頼] → [ユーザー承認] → [実装]
     ↑                                                              │
     └──────────────────── 次のタスクへ ────────────────────────────┘
```

### ラベル定義

| ラベル | 意味 | 担当 |
|--------|------|------|
| `pending-review` | レビュー待ち | ユーザーがレビュー |
| `approved` | 承認済み・実装待ち | AI が実装 |
| `in-progress` | 実装中 | AI |
| `done` | 完了 | クローズ時に付与 |

---

## AI の動作ルール

### 1. プロジェクトを進める時（「進めて」「続けて」等）

1. **承認済みチケットの確認**: `gh issue list --label approved`
2. あれば → 実装開始。`in-progress` に変更し、完了後に `done` でクローズ
3. なければ → 次のタスクを特定し、チケットを作成。`pending-review` を付与してレビュー依頼

### 2. チケット作成時

- GitHub Issue を作成
- ラベル `pending-review` を付与
- ユーザーに「以下のチケットをレビューしてください」と伝える
- 必要に応じて `docs/` の骨子・設計を参照してタスクを分解

### 3. 実装時

- `approved` のチケットのみ実装
- 実装開始時に `in-progress` に変更
- 完了後に `done` を付与してクローズ

---

## ユーザーの操作

### レビュー完了の伝え方

- GitHub の Issue にラベル `approved` を追加
- またはチャットで「#123 承認」「#123 OK」と指定

### トリガー

- 「進めて」「プロジェクトを進めて」「続けて」→ 上記フローを実行
