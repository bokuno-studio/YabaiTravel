# 将来機能: Chatbot によるレース提案

## 概要

会話形式でユーザーのニーズを聞き取り、適したレースを提案する Chatbot 機能の追加を想定。

## 今から気をつけておくこと

### 1. データの構造化を維持する

- レースデータは **DB で構造化**（race_type, event_date, country, location, distance_km, elevation_gain, time_limit, entry_fee 等）されている
- Chatbot が LLM に渡すコンテキストは、この構造化データをそのまま（または要約して）渡すと精度が上がる
- 非構造化の長文だけだと LLM が迷いやすい

### 2. 検索・フィルタのロジックを共通化する

- 一覧のフィルタ条件（レース種別、距離、制限時間等）と、Chatbot が「おすすめ」を出すロジックは **同じ条件**を使うと一貫する
- 可能ならフィルタ条件を関数化し、一覧 UI と Chatbot の両方から呼べるようにしておく

### 3. 国・地域の明示

- `country` カラムを追加済み。Chatbot が「日本で」「海外で」と検索する際に必須

### 4. API の設計

- 現状は Supabase 直接クエリ。将来的に Chatbot が Serverless や API 経由で DB を叩くなら、検索用のエンドポイントを用意しておくと良い
- あるいは Supabase の RPC で検索関数を定義するのもあり

### 5. 会話履歴の扱い

- 「50km くらい」「5月くらい」など曖昧な条件を会話で蓄積する
- セッション単位で会話履歴を保持し、LLM が文脈を理解できるようにする

### 6. リアルタイム性

- データは随時更新される。Chatbot は **毎回最新の DB を参照**する前提が良い（キャッシュしすぎない）

## 実装時の検討事項

- **RAG**: レース数が増えたら、embedding + ベクトル検索で「似たレース」を検索する方が効率的
- **プロンプト**: ユーザー発話から「距離」「場所」「難易度」「予算」等を抽出してフィルタ条件に変換する Few-shot 設計
- **UI**: チャット UI と一覧の連携（提案されたレースをクリックで詳細へ遷移）
